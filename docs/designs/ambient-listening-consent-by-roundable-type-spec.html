Ambient Listening Consent by Roundable Type — Design Spec

Extend Ambient Listening consent configuration and mobile consent modals to support patient, location, and staff rounding — with separate suggested scripts per roundable type and API parameters for mobile clients.

Evolve · Provider Settings CipherRounds Mobile Ambient Listening Patient · Location · Staff

Summary

Today, care providers configure a single Ambient Listening Consent Suggested Script in Provider Settings. Mobile shows a patient-specific consent modal with hardcoded default script lines and the instruction “Please ask the patient for verbal consent before starting the recording.”

This spec splits consent configuration by roundable type (patient, location, staff), updates the mobile consent modal copy per type, and defines the API contract mobile needs to render the correct instruction, suggested script, and action buttons.

  • Evolve: Three type-specific consent script fields, each with its own No Consent Needed Button Enabled checkbox directly below (patient, location, staff).
  • Mobile: Select instruction + script based on script type; stop using hardcoded script lines.
  • Backend: Resolve consent fields on script payloads; extend declined-to-record tracking for staff and location roundables.

Multiline behavior: Each script field accepts free text. Mobile splits on newline characters into separate suggested-script lines (same behavior as today’s single field). Empty lines are ignored.

Location rounding context

Location rounds are often not a 1:1 conversation with the location itself. Common patterns include staff group audits, compliance walk-throughs, and group check-ins. When staff are present, use the suggested script and obtain consent. Organizations often enable Location Ambient Listening No Consent Needed Button Enabled for empty walk-through audits while leaving patient/staff checkboxes off.

Default instruction copy (server-provided)

Script type ambient_listening_consent_instruction default
admitted_patients Please ask the patient for verbal consent before starting the recording.
location / unit If rounding on staff, ask for permission to record.
staff If rounding on a staff member, ask for permission to record.

Staff suggested script (also used as the location script when staff are present):

  • "I'd like to use a digital assistant to help document our conversation."
  • "Is it okay if I record while we talk?"

Location and staff share the same default suggested script when a conversation is involved. Care providers may customize each field independently in Evolve.

Previously declined banner

When the roundable previously declined recording, show the existing banner above the instruction:

  • Title: {RoundableName} previously declined recording (unchanged)
  • Body: Opted out of audio recording on their last round. (unchanged)

Today only patients expose ambient_listening_declined_to_record on the worklist API. Staff and location roundables need the same flag (see API section).

“No Consent Needed” button behavior

Controlled per roundable type by the checkbox under each consent script field. When enabled for a type, that type’s AL modals include a third footer button. Instruction text never tells the rounder to select it.

  • Does not set ambient_listening_declined_to_record on the roundable.
  • Sets recording_verbal_consent_status: "not_needed" on the interaction attempt.
  • Allows recording to start without verbal consent capture.

Layout (when enabled): Three equal-width footer buttons — Consent Given · Declined · No Consent Needed. When disabled, two equal-width buttons — Consent Given · Declined.

API Contract — Mobile Parameters

1. Script payload (Cipher Rounds API v9+)

Sent on script list/show when ambient_listening_enabled is true. Backend resolves values from the care provider based on script type.

Patient script example (patient checkbox on):

{
  "id": "…",
  "type": "admitted_patients",
  "ambient_listening_enabled": true,
  "ambient_listening_consent_instruction": "Please ask the patient for verbal consent before starting the recording.",
  "ambient_listening_consent_suggested_script": "I'd like to use a digital assistant…\n…\nIs it okay if I record our visit today?",
  "ambient_listening_no_consent_needed_button_enabled": true
}

Location script example (location checkbox on):

{
  "id": "…",
  "type": "location",
  "ambient_listening_enabled": true,
  "ambient_listening_consent_instruction": "If rounding on staff, ask for permission to record.",
  "ambient_listening_consent_suggested_script": "I'd like to use a digital assistant…\nIs it okay if I record while we talk?",
  "ambient_listening_no_consent_needed_button_enabled": true
}

Staff script example (staff checkbox off):

{
  "id": "…",
  "type": "staff",
  "ambient_listening_enabled": true,
  "ambient_listening_consent_instruction": "If rounding on a staff member, ask for permission to record.",
  "ambient_listening_consent_suggested_script": "I'd like to use a digital assistant…\nIs it okay if I record while we talk?",
  "ambient_listening_no_consent_needed_button_enabled": false
}

Script payload always uses ambient_listening_no_consent_needed_button_enabled (singular) — backend resolves from the matching per-type CP field.

Field Type When included Mobile usage
ambient_listening_enabled boolean Always on script Gate AL recording bar. Existing field.
ambient_listening_consent_instruction string When AL enabled NEW. Replace hardcoded recordingConsentMessage in consent modal.
ambient_listening_consent_suggested_script string When AL enabled and CP script configured Split on \n into suggested-script lines. Type-resolved from CP. Mobile must decode this field (currently hardcoded in iOS).
ambient_listening_no_consent_needed_button_enabled boolean When AL enabled NEW. Resolved from matching CP field by script type (patient_ / location_ / staff_ prefix). When true, show third footer button for that type only.

Resolution logic (backend)

case script.type
when admitted_patients
  instruction: DEFAULT_PATIENT_INSTRUCTION
  script:      cp.patient_ambient_listening_consent_suggested_script
  no_consent_needed_enabled: cp.patient_ambient_listening_no_consent_needed_button_enabled
when location, unit
  instruction: DEFAULT_LOCATION_INSTRUCTION
  script:      cp.location_ambient_listening_consent_suggested_script
  no_consent_needed_enabled: cp.location_ambient_listening_no_consent_needed_button_enabled
when staff
  instruction: DEFAULT_STAFF_INSTRUCTION
  script:      cp.staff_ambient_listening_consent_suggested_script
  no_consent_needed_enabled: cp.staff_ambient_listening_no_consent_needed_button_enabled
end

# Emit on script JSON as ambient_listening_no_consent_needed_button_enabled

2. Roundable worklist — declined-to-record flag

Roundable Endpoint (existing pattern) Field Notes
Patient GET …/patients/admitted ambient_listening_declined_to_record Exists today. Used to pre-set consent modal declined state.
Staff member GET …/staff_members (index) ambient_listening_declined_to_record NEW. Mirror patient behavior on StaffMember model.
Location / unit GET …/units or location index ambient_listening_declined_to_record NEW. Mirror patient behavior on location roundable model.

3. Interaction draft — consent outcome

Send not_needed only when the third button is visible for that script type (matching CP checkbox on).

POST/PATCH …/interaction_drafts/{attempt_id}

// Consent Given
{ "recording_verbal_consent_status": "given" }

// Declined
{ "recording_verbal_consent_status": "declined" }

// No Consent Needed
{ "recording_verbal_consent_status": "not_needed" }

Backend side effects:

  • given → clear ambient_listening_declined_to_record on the roundable (patient/staff; N/A for location no-op or omit)
  • declined → set ambient_listening_declined_to_record = true where applicable
  • not_needed → do not change declined flags; allow recording without verbal consent
  • Legacy boolean param omitted → do not change declined flag (existing behavior)

Deprecated: boolean-only consent

{ "recording_verbal_consent": true | false }

Still supported for backward compatibility during rollout. New clients should send recording_verbal_consent_status for all roundable types.

4. Evolve provider settings API

GET/PATCH /vue/provider_settings — add three script + three no-consent checkbox fields to serializer and permitted params.

{
  "care_provider": {
    "ambient_listening": true,
    "patient_ambient_listening_consent_suggested_script": "…",

    "location_ambient_listening_no_consent_needed_button_enabled": true,

Backend Implementation Checklist (Banff)

  1. Add three CP fields on CareProvider; data migration copies legacy field → patient field.
  2. Add three CP fields for per-type no-consent checkboxes; wire each under its script field in CareProviderSettings.vue.
  3. Update ProviderSettingsCareProviderSerializer and provider_settings_params.
  4. Update Rounding::ScriptSerializer (and location-scoped variant) to emit ambient_listening_consent_instruction and type-resolved script.
  5. Add ambient_listening_declined_to_record to StaffMember and location roundable models.
  6. Extend interaction draft handler to set/clear declined flag for staff + location (same as patient).
  7. Update Swagger v9 script schema; bump mobile contract tests.

Migration & Backward Compatibility

  • Data: On deploy, copy ambient_listening_consent_suggested_script → patient_ambient_listening_consent_suggested_script where patient field is blank.
  • API read: If type-specific script is blank, fall back to legacy single field (temporary).
  • API write (Evolve): Writing legacy field updates patient field until field is removed.
  • Mobile: If ambient_listening_consent_instruction missing (old API), use platform default by script type table above.
  • Mobile: If suggested script missing, hide suggested-script block; still show instruction + buttons.

Acceptance Criteria

  1. Evolve shows three script fields; each has a No Consent Needed Button checkbox directly below it.
  2. Each roundable type independently shows 2 or 3 footer buttons based on its CP checkbox.
  3. Instruction copy never references the No Consent Needed button.
  4. Patient / location / staff modals show type-appropriate instruction + suggested script.
  5. No Consent Needed (when enabled) sets recording_verbal_consent_status: not_needed and does not mark the roundable as declined.
  6. Consent Given / Declined updates consent status and roundable declined flag for all three roundable types.
  7. Previously declined roundables show the declined banner on next AL round attempt.
  8. Mobile no longer uses hardcoded suggested script lines when API provides a script.
